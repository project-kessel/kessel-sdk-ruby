# AGENTS.md

## Repository Overview

This is `kessel-sdk-ruby`, the official Ruby gRPC client SDK for [Project Kessel](https://github.com/project-kessel). It provides generated protobuf/gRPC bindings for the Kessel Inventory API, a hand-written authentication layer (OAuth2 Client Credentials via OIDC), a gRPC client builder with credential validation, and an RBAC convenience module for workspace operations over HTTP.

The gem is published to RubyGems as `kessel-sdk`. The current version lives in `lib/kessel/version.rb` as `Kessel::Inventory::VERSION`.

## Architecture

### Code Layers

The SDK has a clear three-layer architecture:

1. **Generated layer** (`lib/kessel/inventory/v*/**/*_pb.rb`, `lib/google/**/*`, `lib/buf/**/*`) -- Protobuf message classes and gRPC service stubs generated by `buf generate` from `buf.build/project-kessel/inventory-api`. Never hand-edit these files.

2. **Core hand-written layer** (`lib/kessel/auth.rb`, `lib/kessel/grpc.rb`, `lib/kessel/inventory.rb`, `lib/kessel/version.rb`) -- Authentication, gRPC credential wiring, and the `ClientBuilder` base class. These files are subject to RuboCop and have corresponding RBS type signatures in `sig/kessel/`.

3. **Convenience layer** (`lib/kessel/rbac/v2.rb`, `v2_helpers.rb`, `v2_http.rb`) -- Higher-level helpers built on top of V1beta2 protobuf types for RBAC workspace operations. Uses both gRPC (via `streamed_list_objects`) and HTTP (via `Net::HTTP` for workspace fetches).

### Module Namespace Map

```
Kessel::Inventory          -- client_builder_for_stub, ClientBuilder base class
Kessel::Inventory::V1      -- Health check service only
Kessel::Inventory::V1beta1 -- Legacy typed K8s resources/relationships (Resources, Relationships sub-modules)
Kessel::Inventory::V1beta2 -- Current unified inventory service (KesselInventoryService)
Kessel::Auth               -- OAuth2ClientCredentials, AuthRequest interface, OIDCDiscoveryMetadata
Kessel::GRPC               -- oauth2_call_credentials helper
Kessel::RBAC::V2           -- Workspace helpers, principal/role/subject factories, list_workspaces enumerator
```

New SDK features should target `V1beta2` unless explicitly working with legacy services.

### Service Wiring Pattern

Every gRPC service module must follow this exact pattern to be properly wired:

```ruby
# lib/kessel/inventory/v1betaN.rb
require 'kessel/inventory'
require 'kessel/inventory/v1betaN/some_service_services_pb'

include Kessel::Inventory

module Kessel::Inventory::V1betaN
  module KesselSomeService
    ClientBuilder = ::Kessel::Inventory.client_builder_for_stub(Stub)
  end
end
```

The `include Kessel::Inventory` at the top level is required for `client_builder_for_stub` to resolve. The `Stub` constant comes from the generated `*_services_pb.rb`. When adding a new service, also add the `ClientBuilder` constant declaration in `sig/kessel/inventory.rbs`.

### Version Naming Convention

Protobuf package `kessel.inventory.v1beta2` maps to Ruby module `Kessel::Inventory::V1beta2` -- note lowercase `beta` (not `V1Beta2`). This follows the protobuf-to-Ruby naming convention from `buf generate`.

## Cross-Cutting Conventions

### File Headers

Every hand-written Ruby file must start with `# frozen_string_literal: true`. This is a security and correctness requirement -- it prevents accidental string mutation in header values and token strings.

### Single Runtime Dependency

The gemspec declares only one runtime dependency: `grpc >= 1.73.0`. The `openid_connect` gem is intentionally optional and lazy-loaded via `require 'openid_connect'` inside `check_dependencies!`. Do not add `openid_connect` as a hard runtime dependency.

### Generated vs. Hand-Written Code Boundary

- **Generated files** live under `lib/kessel/inventory/v*/**/*`, `lib/google/**/*`, and `lib/buf/**/*`. They are excluded from RuboCop, YARD, and should not be tested directly (only verify they load without error).
- **Hand-written files** are everything else under `lib/kessel/`. These are subject to RuboCop limits: method length 25, class length 150, cyclomatic complexity 10, line length 120.
- When `buf generate` runs, it overwrites all generated files. The GitHub Actions workflow `buf-generate.yml` does this every 6 hours and creates a PR on the `buf-generate-update` branch.

### RBS Type Signatures

Hand-written RBS files in `sig/kessel/` cover only the hand-written code. When modifying or adding hand-written modules, update the corresponding `.rbs` file. Generated protobuf code has no RBS signatures. The `Steepfile` checks `lib/` against `sig/` using `rbs_collection` for the `grpc` gem types.

### Builder Pattern

The `ClientBuilder` uses a fluent API where authentication methods (`insecure`, `authenticated`, `unauthenticated`, `oauth2_client_authenticated`) return `self` and must be followed by `.build`. Each method calls `validate_credentials` immediately, so invalid configurations fail at configuration time, not at request time.

`client_builder_for_stub` creates a new `Class` dynamically via `Class.new(ClientBuilder)`. The SDK caches these as constants (e.g., `KesselInventoryService::ClientBuilder`). Do not call `client_builder_for_stub` repeatedly at runtime.

### Error Conventions

- Custom exceptions live under `Kessel::Auth` and inherit from `StandardError`.
- `ClientBuilder` and `RBAC::V2` use bare `raise "message"` (`RuntimeError`) for validation errors.
- gRPC errors (`GRPC::BadStatus` subclasses) are never wrapped -- they propagate directly to callers.
- Rescue `StandardError`, never `Exception`, in new code. Re-raise after logging with bare `raise`.

### Example File Conventions

- Examples live in `examples/` with their own `Gemfile` (includes `dotenv` and `openid_connect`).
- Examples are excluded from RuboCop, specs, and coverage.
- Newer examples (preferred pattern) use a class with `class << self` and `rescue StandardError => e` with bare `raise`. Older examples use top-level `include` and `rescue Exception => e`.
- Use `ENV.fetch('VAR_NAME', nil)` for environment variable access, not `ENV['VAR_NAME']`.

## Common Pitfalls

### Do not hand-edit generated protobuf files

Any change under `lib/kessel/inventory/v*/**/*`, `lib/google/**/*`, or `lib/buf/**/*` will be silently overwritten by the next `buf generate` run (which happens automatically every 6 hours via CI). If a protobuf change is needed, it must go upstream in the `.proto` definitions at `buf.build/project-kessel/inventory-api`.

### Do not make `openid_connect` a hard dependency

The SDK is designed so consumers who do not need OAuth are not forced to install `openid_connect` and its transitive dependencies. The `check_dependencies!` method in the `OAuth2ClientCredentials` constructor handles this at runtime.

### Do not build gRPC clients per-request

`ClientBuilder.build` creates a new `Stub` instance each time. The underlying gRPC channel manages its own HTTP/2 connection pool. Build once at application startup and reuse.

### Do not add blanket gRPC error wrapping

The SDK intentionally does not catch or wrap `GRPC::BadStatus`. This is by design -- it is a thin client layer, and callers handle gRPC errors in their own code.

### Do not reduce the token expiration window

`EXPIRATION_WINDOW = 300` (5 minutes) ensures tokens are refreshed proactively before they expire during in-flight requests. Reducing this value creates race conditions.

### Do not bypass credential validation order

Each `ClientBuilder` authentication method calls `validate_credentials` immediately. This ensures insecure channels cannot be composed with call credentials. Moving validation to `build` time would allow invalid configurations to exist silently.

### RuboCop and bundle-audit are non-blocking in CI

Both run with `continue-on-error: true` in the CI workflow. Fix violations even though they do not block merges. Only `rspec` failures block CI.

## Release Process

- Version lives in `lib/kessel/version.rb` as `Kessel::Inventory::VERSION`.
- SDK versions follow SemVer independently from other Kessel language SDKs.
- Release branches use the naming convention `release/${VERSION}`.
- Run `buf generate` before releasing to pick up any upstream protobuf changes.
- The gemspec requires `rubygems_mfa_required = 'true'` -- MFA is enforced for all gem publishes.

## CI/CD

- **CI workflow** (`.github/workflows/ci.yml`): Runs on push/PR to `main`. Tests Ruby 3.3 and 3.4 on ubuntu-latest. Steps: `bundle install`, `rspec`, `rubocop --parallel` (non-blocking), `bundle-audit` (non-blocking). Also builds and installs the gem.
- **buf-generate workflow** (`.github/workflows/buf-generate.yml`): Runs every 6 hours and on manual dispatch. Regenerates protobuf code and auto-creates a PR on `buf-generate-update` branch.
- **Dependabot** (`.github/dependabot.yml`): Daily checks for both `bundler` and `github-actions` ecosystems.

## PR Expectations

- All specs must pass on Ruby 3.3 and 3.4.
- Fix RuboCop violations before merging (even though CI does not block on them).
- If modifying hand-written code, update the corresponding RBS type signature in `sig/kessel/`.
- If adding a new example, add it to the README examples list.
- If adding a new service version, follow the service wiring pattern exactly and add the `ClientBuilder` constant to `sig/kessel/inventory.rbs`.
- Never commit `.env` files or credentials. The `.gitignore` covers `.env`, `.env.local`, `.env.*.local`, `credentials.yml`, and `secrets.yml`.
