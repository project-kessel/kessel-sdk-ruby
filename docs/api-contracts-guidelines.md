# API Contracts Guidelines

## Protobuf Source and Code Generation

- All protobuf definitions are sourced from the remote BSR module `buf.build/project-kessel/inventory-api`. No `.proto` files exist in this repo.
- Run `buf generate` to regenerate Ruby code. This uses `buf.gen.yaml` which configures two remote plugins: `buf.build/protocolbuffers/ruby` for message classes and `buf.build/grpc/ruby` for service stubs, both outputting to `lib/` with `paths=source_relative` and `include_imports: true`.
- A GitHub Actions workflow (`.github/workflows/buf-generate.yml`) runs `buf generate` every 6 hours and auto-creates a PR on the `buf-generate-update` branch if files changed.
- Never hand-edit files under `lib/kessel/inventory/v1*/`, `lib/google/`, or `lib/buf/`. These are all generated and marked with `# Generated by the protocol buffer compiler. DO NOT EDIT!`.

## Module and Namespace Structure

- The protobuf package `kessel.inventory.v1beta2` maps to Ruby module `Kessel::Inventory::V1beta2`. The naming convention uses lowercase `beta` (e.g., `V1beta2` not `V1Beta2`).
- Three API versions coexist: `V1` (stable health checks only), `V1beta1` (legacy typed resource/relationship services), and `V1beta2` (current unified inventory service). New SDK features target `V1beta2`.
- The entry point `lib/kessel-sdk.rb` requires all three version modules, plus `Kessel::Auth`, `Kessel::GRPC`, `Kessel::RBAC::V2`, and the RBAC helper modules (`v2_helpers`, `v2_http`).
- V1beta1 organizes services under sub-modules `Resources` and `Relationships`, while V1beta2 consolidates into a single `KesselInventoryService`.

## Service Wiring Pattern

Each versioned module file (e.g., `lib/kessel/inventory/v1beta2.rb`) must: (1) require the generated `*_services_pb.rb`, (2) call `client_builder_for_stub(Stub)` inside the service module to create a `ClientBuilder` constant.

The `Stub` constant is auto-generated in `*_services_pb.rb` via `Stub = Service.rpc_stub_class`.

When adding a new versioned service, follow this exact pattern:
```ruby
module Kessel::Inventory::V1betaN
  module KesselSomeService
    ClientBuilder = ::Kessel::Inventory.client_builder_for_stub(Stub)
  end
end
```

## ClientBuilder Fluent API

- Four authentication modes exist: `oauth2_client_authenticated`, `authenticated`, `unauthenticated`, and `insecure`. Each returns `self` and must be followed by `.build`.
- Insecure mode sets `channel_credentials` to `:this_channel_is_insecure`. Combining call credentials with insecure channel raises a validation error.
- When no channel credentials are explicitly set, `build` defaults to `GRPC::Core::ChannelCredentials.new` (TLS).
- The `client_builder_for_stub` method dynamically creates a `Class.new(ClientBuilder)` subclass with the stub class stored as a class-level instance variable.

## Authentication Contracts

- The `Kessel::Auth` module provides OAuth2 Client Credentials flow via `OAuth2ClientCredentials`. The `openid_connect` gem is an optional dependency; its absence raises `OAuthDependencyError` at initialization.
- Token caching uses a `Mutex` for thread safety with a 5-minute expiration window (`EXPIRATION_WINDOW = 300`). Default token lifetime is 1 hour (`DEFAULT_EXPIRES_IN = 3600`).
- The `AuthRequest` interface defines a single method `configure_request(request)` that sets the `authorization` header on `Net::HTTPRequest` objects. All custom auth adapters must implement this interface.
- For gRPC auth, use `Kessel::GRPC#oauth2_call_credentials` which wraps tokens as `GRPC::Core::CallCredentials`. For HTTP auth (RBAC), use `Kessel::Auth#oauth2_auth_request` which returns an `OAuth2AuthRequest`.

## RBAC V2 Module (Hand-Written Layer)

- `Kessel::RBAC::V2` is the hand-written convenience layer in `lib/kessel/rbac/` (three files: `v2.rb`, `v2_helpers.rb`, `v2_http.rb`). It builds on top of `V1beta2` protobuf types.
- Helper methods (`workspace_type`, `principal_resource`, `principal_subject`, etc.) construct protobuf message objects with `reporter_type: 'rbac'` hardcoded.
- Principal resource IDs use the format `"#{domain}/#{id}"`.
- The `list_workspaces` method returns a lazy `Enumerator` that auto-paginates via `continuation_token` with `DEFAULT_PAGE_LIMIT = 1000`.
- HTTP workspace fetch (`fetch_workspace`) requires `x-rh-rbac-org-id` header and uses `WORKSPACE_ENDPOINT = '/api/rbac/v2/workspaces/'`.
- When an `http_client` is provided, it must match the host and port of the endpoint URI.

## Request/Response Patterns (V1beta2)

- Check operations use a triplet: `object` (ResourceReference), `relation` (string), `subject` (SubjectReference). `CheckSelf` omits the subject (inferred from auth context).
- `ResourceReference` requires `resource_type` (regex `^[A-Za-z0-9_]+$`), `resource_id` (non-empty), and optional `reporter` (ReporterReference).
- `CheckForUpdate` provides strongly consistent checks (no `consistency` field); `Check` accepts an optional `Consistency` oneof with `minimize_latency`, `at_least_as_fresh` (token), or `at_least_as_acknowledged`.
- Bulk operations (`CheckBulk`, `CheckSelfBulk`, `CheckForUpdateBulk`) wrap items in a `repeated` field with validation `min=1, max=1000`. Responses use a `pairs` field containing `CheckBulkResponsePair` with either `item` or `error` (google.rpc.Status).
- `ReportResourceRequest` requires `type`, `reporter_type`, `reporter_instance_id`, `representations`, and `write_visibility` enum (`WRITE_VISIBILITY_UNSPECIFIED`, `MINIMIZE_LATENCY`, `IMMEDIATE`).
- Streaming RPCs (`StreamedListObjects`, `StreamedListSubjects`) use server-side streaming with `RequestPagination` (limit + continuation_token).

## Allowed Enum Convention

- V1beta2 uses a shared top-level `Allowed` enum: `ALLOWED_UNSPECIFIED`, `ALLOWED_TRUE`, `ALLOWED_FALSE`.
- V1beta1 uses per-response nested enums (`CheckResponse::Allowed`).

## RBS Type Signatures

- Hand-written RBS files live in `sig/kessel/` and cover only the hand-written Ruby code (`auth.rbs`, `grpc.rbs`, `inventory.rbs`, `rbac.rbs`, `version.rbs`). Generated protobuf code has no RBS signatures.
- The `Steepfile` checks `lib/` against `sig/` with `rbs_collection` for `grpc` types.
- When adding a new hand-written module, create a corresponding `.rbs` file in `sig/kessel/`. When adding a new versioned service, add a `ClientBuilder` constant declaration in `sig/kessel/inventory.rbs`.

## Versioning and Releases

- SDK version lives in `lib/kessel/version.rb` as `Kessel::Inventory::VERSION`. The gemspec reads this constant.
- SDK versions follow SemVer independently from other Kessel language SDKs.
- Required Ruby version: `>= 3.3`. Only runtime dependency is `grpc >= 1.73.0`.

## HTTP API Route Convention

- V1 HTTP routes: `/api/kessel/v1/{method}` (e.g., `/api/kessel/v1/livez`)
- V1beta1 HTTP routes: `/api/inventory/v1beta1/{domain}/{method}` (e.g., `/api/inventory/v1beta1/authz/check`)
- V1beta2 HTTP routes: `/api/kessel/v1beta2/{method}` (e.g., `/api/kessel/v1beta2/check`)
- RBAC V2 REST endpoint: `/api/rbac/v2/workspaces/`
